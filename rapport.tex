\chapter{Introduction}

Ce projet fait partie de l'UE IA pour les systèmes cyber-physiques en Master 1. L'objectif est de faire de la classification de séries temporelles. Ces données représentent des traces d'activités de processeur (CPU), de disque et de mémoire vive (RAM). 

Le but est de pouvoir identifier automatiquement de quel type d'activité il s'agit à partir de traces d'exécution.

\section{Objectifs du projet}
Le travail consiste à tester et comparer trois approches différentes :
\begin{itemize}
    \item Les réseaux de neurones récurrents de type LSTM.
    \item Les réseaux de neurones convolutionnels (CNN 1D).
    \item Les modèles plus récents basés sur le mécanisme d'attention (Transformers).
\end{itemize}

Puis proposer des pistes susceptibles de répondre au problème identifié afin de déterminer si les séquences de traces d’exécution proviennent d’un navigateur web, d’un compilateur, d’un logiciel de traitement de texte, etc.

\section{Préparation des Données}
Pour que les modèles puissent apprendre efficacement, il faut appliquer plusieurs étapes aux données :

\begin{itemize}
    \item \textbf{Sélection des données} : Nous retenons les 4 variables les plus intéressantes :
    \begin{enumerate}
        \item \texttt{CpuALL\_usage} : Charge globale du processeur.
        \item \texttt{memory\_usage\_perce} : Occupation de la RAM.
        \item \texttt{read\_rate\_during\_time} : Vitesse de lecture disque.
        \item \texttt{write\_rate\_during\_time} : Vitesse d'écriture disque.
    \end{enumerate}
    \item \textbf{Normalisation} : Afin de pouvoir comparer les valeurs plus facilement, nous les ramenons sur un intervalle commun entre 0 et 1 grâce à \textit{MinMaxScaler}. Cela permet l'unicité des données, en évitant que certaines valeurs soient exprimées en pourcentage comme l'activité CPU et d'autres en octets comme la mémoire ; on pourra donc se les représenter dans un même graphe afin de permettre un meilleur apprentissage des modèles.
    \item \textbf{Fenêtrage} : Les données sont découpées en séquences de 50 unités de temps. Pour les modèles LSTM et CNN, c'est crucial afin qu'ils capturent l'activité sur un intervalle de temps plutôt qu'à un instant T. C'est ce bloc de 50 points qui sera donné en entrée aux modèles.
    \item \textbf{Labels} : Pour pouvoir entraîner nos modèles, il faut étiqueter nos données ; pour cela, nous avons déterminé que \textbf{CPU = 0, DISQUE = 1 et MÉMOIRE = 2}.
\end{itemize}

\chapter{Modèle 1 : LSTM (Long Short-Term Memory)}

\section{Fonctionnement}
Le LSTM est un réseau de neurones utilisé pour traiter des données séquentielles. Ce type de réseau a la particularité de pouvoir retenir des informations sur le long terme. Cela va être très utile pour reconnaître un pic d'activité CPU, car il faut analyser dans un contexte temporel, ici sur un intervalle de 50 unités de temps.

Son fonctionnement consiste à décider quelles informations passées doivent être conservées, mises à jour ou oubliées, ce qui nous permettra de reconnaître aisément une trace qui s'étale dans le temps (ex : une lecture disque).

\section{Architecture}
Nous avons construit un modèle avec les caractéristiques suivantes :
\begin{itemize}
    \item \textbf{Entrée} : Des tenseurs sous la forme (50, 4).
    \item \textbf{Couche LSTM} : 64 unités pour capturer la dynamique des 4 variables.
    \item \textbf{Dropout (20\%)} : Pour éviter le sur-apprentissage, 20\% des neurones sont désactivés aléatoirement durant l'entraînement.
    \item \textbf{Couche Dense} : 32 neurones ReLU pour la prise de décision.
    \item \textbf{Sortie} : 3 neurones Softmax pour distribuer une probabilité pour chacune des 3 classes (CPU, Disque, Mémoire).
\end{itemize}

L'entraînement est réalisé sur 15 epochs et batch size de 64.

Avec les paramètres suivants :
\begin{itemize}
    \item \textbf{Optimiseur} : Adam
    \item \textbf{Fonction de perte} : \texttt{sparse\_categorical\_crossentropy}
\end{itemize}

\chapter{Modèle 2 : CNN 1D (Convolutional Neural Network)}

\section{Fonctionnement}
Le réseau de neurones convolutionnel (CNN) peut être utilisé sur des séries temporelles avec des convolutions 1D. Au lieu de regarder une série dans son intégralité, le modèle utilise des filtres pour détecter des motifs (pics, transitions brusques, etc.) sur la série temporelle. 

L'avantage du CNN par rapport au LSTM est sa capacité à repérer des signatures de formes précises sans avoir besoin de les mémoriser.

\section{Architecture}
Pour le modèle CNN 1D, nous avons défini l'architecture suivante :
\begin{itemize}
    \item \textbf{1re Couche Conv1D} : 64 filtres, une taille de noyau (kernel) de 3 ; pour repérer les motifs simples.
    \item \textbf{2e Couche Conv1D} : 128 filtres, une taille de noyau (kernel) de 3 ; pour repérer les combinaisons des motifs repérés par la couche précédente.
    \item \textbf{MaxPooling1D} : Réduit la dimension temporelle par 2, pour ne garder que les valeurs les plus importantes.
    \item \textbf{Dropout (20\%)} : Pour éviter le sur-apprentissage, 20\% des neurones sont désactivés aléatoirement durant l'entraînement.
    \item \textbf{Couche Dense} : 32 neurones ReLU pour la prise de décision.
    \item \textbf{Sortie} : 3 neurones Softmax pour distribuer une probabilité pour chacune des 3 classes (CPU, Disque, Mémoire).
\end{itemize}

L'entraînement est réalisé sur 15 epochs et batch size de 64.

Avec les paramètres suivants :
\begin{itemize}
    \item \textbf{Optimiseur} : Adam

    \item Test : 0,005 ou 0,0005.
    \item Observation : À 0,0005, les erreurs augmentent lors de l'évaluation ; à 0,005, aucun changement significatif n'est relevé.
\end{itemize}
\begin{itemize}
    \item \textbf{GAP} :
    \item Test : Remplacement de la couche \texttt{Flatten()} par \texttt{GlobalAveragePooling1D()}.
    \item Observation : Pas de changement significatif.
\end{itemize}
\begin{itemize}
    \item \textbf{Poids des erreurs} : Rendre les erreurs sur la classe mémoire plus punitives pour augmenter l'attention du modèle sur celle-ci.

Ces tests ont permis de définir la meilleure configuration, privilégiant la stabilité et la régularisation : un modèle à 2 ou 3 blocs avec \texttt{BatchNormalization}, un \textit{Dropout} de 0,3 et un noyau de taille 3 ou 5.

L'amélioration porte sur la classe \textbf{Mémoire} ; en rendant les erreurs plus punitives, le rappel passe à \textbf{0.69}, mais la précision générale diminue. La confusion avec le CPU est toujours présente. Les filtres de convolution semblent mieux repérer certaines signatures liées aux accès RAM que le mécanisme de mémoire du LSTM.

\chapter{Modèle 3 : Time Series Transformer}

\section{Fonctionnement}
Le Transformer est l'architecture qui utilise un mécanisme d'\textbf{Attention}. Le modele analyse simultanément tous les points de la fenetre et calcule un score d’importance par rapport à tous les autres points de la séquence

\section{Architecture}
L'architecture implémentée repose sur un bloc de codage, puis d'une tête de classification :

\begin{itemize}
    \item \textbf{Multi-Head Attention} : 4 têtes d'attention. Chaque tête apprend reperer un types de motifs différents.
    \item \textbf{Layer Normalization} : Stabilise l'apprentissage 
    \item \textbf{Feed-Forward Network (Conv1D)} : Deux couches de convolution avec un noyau de taille 1 qui fonctionne comme un réseau de neurones, pour les caractéristiques plus complexes.
    \item \textbf{Connexions Résiduelles} : Evite la dégradation des gradients et facilite l'entraînement du modèles.
    \item \textbf{Global Average Pooling 1D} : Compresse la séquence en un vecteur unique., qui sera utilisé pour la décision finale.
    \item \textbf{Couches Denses} : Une série de couches de neurone termine le traitement avant le \textbf{Softmax} à 3 classes.
\end{itemize}

\section{Introduction de la variable "Energy Usage"}
Nous avons utilisé quatre variables (CPU, RAM, Lectures et Écritures Disque). Etant donnes que tout les modele avasi du mal sur la difference entre CPU et MEMOIRE, peutre que le probleme vien du manque de donee. Afin de tester les modèles et d'améliorer la distinction entre les classes, nous avons décidé d'ajouter une cinquième variable : \texttt{Energy\_usage\_during\_time}.
Nous pensont que la consommation énergétique suit des signatures très spécifiques selon le composant sollicité.
\section{Résultats et Comparaisons}
Nous avons réentraîné nos trois modele (LSTM, CNN 1D et Transformer) avec cette donnée suplémentaire. Les résultats montrent une amélioration de la précision sur tous les modèles.

\section{Analyse de l'apport de l'Énergie}
L'ajout de cette variable a permis de lever une grande partie des ambiguïtés, notamment :
\begin{itemize}
    \item \textbf{Meilleure séparation CPU/Mémoire} : Le profil énergétique d'une intensité de calcul (CPU) est désormais plus clairement différencié des cycles de gestion mémoire.
    \item \textbf{Stabilité du rappel} : Le rappel pour la classe Mémoire, qui stagnait autour de 0.60, a connu une progression significative.
    \item \textbf{Rapidité de convergence} : Les modèles atteignent leurs scores maximaux en moins d'époques, car l'information supplémentaire rend la "signature" de chaque classe plus discriminante.
\end{itemize}

\section{Conclusion}
L'intégration de la variable \texttt{Energy\_usage} confirme un principe fondamental de l'apprentissage automatique : la qualité et la pertinence des données d'entrée (*Feature Engineering*) sont souvent plus déterminantes que la complexité des algorithmes eux-mêmes. Même un modèle plus simple obtient de meilleurs résultats avec des descripteurs plus riches.
Cette étape valide notre choix de caractéristiques et nous permet de sélectionner la version finale de nos modèles pour la synthèse globale.

\section{Perspectives : Vers une application réelle}
Pour passer de l'identification de "micro-tâches" (50 secondes) à l'identification d'une application entière (ex: Chrome, GCC), nous proposons une \textbf{approche hiérarchique} :

\begin{enumerate}
    \item \textbf{Niveau 1} : Notre CNN 1D actuel étiquette chaque bloc de 50 secondes. On obtient une séquence d'étiquettes du type $[CPU, CPU, DISK, MEMORY, \dots]$.
    \item \textbf{Niveau 2} : Un second modèle (léger) analyse cette "macro-séquence" d'étiquettes pour reconnaître la signature d'un logiciel complet.
\end{enumerate}
